# Деплой на хост

Ниже — **универсальная** схема; конкретный путь на диске и пользователь (`root` / `deploy`) зависят от вашей инфраструктуры.

## Вариант A: Docker (рекомендуется)

На сервере в каталоге с репозиторием:

```bash
git pull origin main
cp -n .env.example .env    # если .env ещё нет; иначе не перезаписывать
# отредактируй .env (S3, maintenance, URL картинок)
docker compose build
docker compose up -d
```

- Образ собирается из **`Dockerfile`**: `npm ci --omit=dev`, `CMD ["node", "server.js"]`.
- **`docker-compose.yml`** монтирует `env_file: .env` и пробрасывает порт **3000**.

Обновление после `git pull`:

```bash
docker compose up -d --build
```

Если менялись только JS без зависимостей, иногда достаточно пересобрать образ или перезапустить контейнер — ориентируйся на изменения в `package.json` / lockfile.

## Вариант B: Node без Docker

```bash
git pull origin main
npm ci --omit=dev
# systemd unit / pm2 / screen — запуск node server.js с рабочей директорией = корень репо
PORT=3000 node server.js
```

Убедись, что процесс перезапускается при деплое и при падении.

## Переменные окружения

Обязательно иметь на сервере свой **`.env`** (не из git). Описание переменных: [05-environment-variables.md](05-environment-variables.md).

## HTTPS и WebSocket

В продакшене обычно перед приложением стоит **nginx** (или другой reverse proxy) с TLS. WebSocket должен проксироваться (`Upgrade`, `Connection`). Конфиг прокси не входит в этот репозиторий.

## Чеклист релиза

1. `git pull` на нужной ветке / теге.
2. Проверить миграции данных не требуются (состояние монополии в S3 — осторожно с несовместимыми форматами).
3. Пересборка / рестарт.
4. Дымовой тест: открыть главную, создать комнату, один режим игры.
5. При включённом maintenance — проверить bypass.
